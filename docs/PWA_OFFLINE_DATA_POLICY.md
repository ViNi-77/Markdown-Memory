# PWA オフラインデータ方針

Markdown Memory の初期PWAでは、ホーム画面追加、起動、スマホ閲覧、オフライン時の案内までを採用します。

ログイン後のワークスペース、非公開Markdown本文、共有ページ、API応答を端末へ永続保存するオフライン閲覧/編集は採用しません。

## 現在の判断

| 項目                     | 判断              | 理由                                                                |
| ------------------------ | ----------------- | ------------------------------------------------------------------- |
| 非公開Markdown本文       | 端末保存しない    | 端末紛失、共有端末、ログアウト後残存のリスクが大きい                |
| ログイン後ワークスペース | 端末保存しない    | フォルダ名、ファイル名、本文、共有状態を含むため                    |
| 共有ページ               | 端末保存しない    | URLを知る人が読める仕様でも、受信側端末に残す判断は別問題として扱う |
| API応答                  | 端末保存しない    | 本文、メタデータ、認証状態が混ざる可能性がある                      |
| オフライン閲覧           | 初期PWAでは見送る | 明示同意、暗号化、削除導線、同期方針が揃ってから判断する            |
| オフライン編集           | 初期PWAでは見送る | 競合解決、保存失敗時の復旧、同期順序の設計が必要                    |

この判断は「PWA化をしない」という意味ではありません。安全なアプリシェルだけを端末に置き、Markdown本文はオンライン接続時にサーバーから取得する設計を保ちます。

## 端末に置いてよいもの

| 保存先               | 許可する内容                                                   |
| -------------------- | -------------------------------------------------------------- |
| Service Worker Cache | `/demo`、`/offline`、PWAアイコン、Next.jsの静的アセット        |
| Browser memory       | 画面表示中の編集中テキストやデモ状態。再読み込みで失われてよい |
| Auth.js Cookie       | 認証に必要なCookie。Auth.jsの管理対象として扱う                |
| `localStorage`       | ユーザーが明示保存した BYOK Gemini API キー                    |
| Vercel / Neon / DB   | ログイン後のMarkdown本文、フォルダ、共有状態の正本             |

BYOK Gemini API キーは Markdown 本文とは別の扱いです。Markdown本文を `localStorage`、`sessionStorage`、`IndexedDB`、Cache Storage に保存する実装は追加しません。

## 端末に置かないもの

- ログイン後ワークスペースのHTML
- 非公開Markdown本文
- フォルダ名、ファイル名、本文メタデータのAPI応答
- `/api/*` のレスポンス
- `/share/*` の共有ページ本文
- `/view/*` の全画面閲覧本文
- `/login` と認証まわりの画面
- AI要約、整形結果、AIへ渡すためのプロンプト本文

Service Worker はこれらのパスをキャッシュ対象外にします。ブラウザ自体のHTTPキャッシュを完全に制御するものではないため、機密性の高い本文を扱う場合は共有端末でログインしない運用を前提にします。

## 将来オフライン閲覧を採用する条件

オフライン閲覧を検討する場合は、以下をすべて満たしてから別Phaseで実装します。

- ユーザーがファイル単位で明示的に「この端末に保存」を選べる
- 端末保存される内容、リスク、削除方法を保存前に説明する
- 端末内データを暗号化し、BYOK Gemini API キーと同じ保存先・同じ意味の秘密として扱わない
- ログアウト時に端末保存データを削除する
- 設定画面から端末保存データを全削除できる
- どのファイルが端末保存中か一覧で確認できる
- 共有ページは受信側端末に自動保存しない
- E2E で保存対象、非対象、削除導線、ログアウト時削除を固定する
- 端末紛失、共有端末、ブラウザプロファイル共有時の注意を docs に明記する

## 将来オフライン編集を採用する条件

オフライン編集は閲覧よりリスクと複雑さが高いため、初期PWAでは採用しません。検討する場合は、少なくとも以下を決めます。

- 同じファイルを複数端末で編集した場合の競合解決
- オフライン保存キューの可視化
- 同期失敗時のリトライと手動復旧
- サーバー上の最新版と端末上の編集差分の比較方法
- 削除、リネーム、フォルダ移動の競合処理
- 共有公開状態をオフラインで変更できるかどうか

## 実装時のチェック

PWAまたは保存まわりを変更するPRでは、以下を確認します。

- [ ] `public/sw.js` が `/`, `/login`, `/api/*`, `/share/*`, `/view/*` をキャッシュしない
- [ ] Markdown本文を `localStorage`、`sessionStorage`、`IndexedDB`、Cache Storage に保存する変更がない
- [ ] BYOK Gemini API キーの保存と Markdown本文の保存を混同していない
- [ ] 共有ページを offline-ready にしていない
- [ ] オフライン時は `/offline` の案内へ落ちる
- [ ] 新しく端末保存するデータがある場合、この方針を更新している

## 現時点の結論

Phase 7H の結論として、Markdown Memory は「オンライン前提のPWA」として進めます。

ホーム画面追加とスマホ読書体験は磨きますが、非公開Markdown本文のオフライン閲覧/編集は、明示同意、暗号化、削除導線、同期方針が揃うまで採用しません。
