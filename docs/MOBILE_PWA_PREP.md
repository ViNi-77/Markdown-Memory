# モバイル / PWA 準備メモ

Markdown Memory の長期ゴールは、PCが手元にない場面でもスマホからMarkdownを確認できる状態にすることです。

このドキュメントでは、いきなりPWA化やネイティブアプリ化へ進む前に満たすべき前提を整理します。

## 現在の位置づけ

現在のUIは、デスクトップの4ペイン構成を主軸にしています。スマホ幅ではフォルダペインを隠し、ファイル一覧・本文・詳細を横スクロールと下部ナビで確認する前段階の作りです。

PWAの下地として、manifest、アイコン、オフラインページ、Service Worker は追加済みです。ただし、スマホ専用UIやPWAインストール体験はまだ本採用していません。

## 先に守ること

- スマホ幅でも `/demo` が開ける
- ファイル一覧からMarkdown本文へ移動できる
- ファイル一覧から詳細ペインへ移動できる
- Markdown本文が読める
- ログイン、共有、編集まわりのセキュリティ仕様を変えない
- 非公開Markdown本文、API応答、共有ページをService Workerでキャッシュしない

## PWA下地の現在仕様

| 項目             | 現在の実装                                                         |
| ---------------- | ------------------------------------------------------------------ |
| manifest         | `/manifest.webmanifest`                                            |
| アイコン         | `/icons/icon.svg`、`/icons/maskable-icon.svg`                      |
| オフライン画面   | `/offline`                                                         |
| Service Worker   | `public/sw.js`                                                     |
| キャッシュ対象   | `/demo`、`/offline`、アイコン、Next.jsの静的アセット               |
| キャッシュ対象外 | `/`、`/api/*`、`/share/*`、`/view/*`、`/login`、非公開Markdown本文 |

## 自動確認

Playwright でスマホ幅の最低限の閲覧導線を確認します。

```bash
npm run test:e2e
```

対象は `/demo` です。保存や共有を伴わないため、本番DBやOAuth設定に依存しません。

## PWA化前に決めること

| 項目           | 方針                                                                                             |
| -------------- | ------------------------------------------------------------------------------------------------ |
| オフライン閲覧 | 非公開Markdown本文のキャッシュは未採用。導入する場合は暗号化・明示同意・削除導線とセットで判断。 |
| オフライン編集 | 競合解決が必要なため、初期PWAでは見送る候補。                                                    |
| ローカル保存   | BYOKキーとMarkdown本文を同じ扱いにしない。本文保存は慎重に判断。                                 |
| 共有リンク     | PWAでもURLを知る人が読める仕様は変えない。                                                       |
| 通知           | 初期段階では不要。                                                                               |
| ストア配信     | Web PWAの安定後に検討。                                                                          |

## 次のUI候補

スマホ専用UIを作る場合は、4ペイン横スクロールから、以下のような段階式ナビゲーションに変える案が自然です。

1. ファイル一覧
2. Markdown本文
3. 詳細・共有・AI操作

この形にすると、スマホでは「読む」を優先し、編集や管理操作は必要なときだけ開く設計にできます。

## 次に確認すること

- iPhone Safari / Android Chrome で `/demo` を開き、下部ナビで一覧・本文・詳細へ移動できるか
- ホーム画面追加時のアイコンとアプリ名が自然か
- オフライン時に `/offline` が表示されるか
- ログイン後のワークスペースや共有ページが端末に残らない方針で問題ないか
